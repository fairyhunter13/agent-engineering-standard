# Snapshot Testing Hygiene

## When to use
Apply this skill when snapshot tests are updated blindly (`--updateSnapshot` run without review), when snapshot diffs are hundreds of lines long, when CI automatically updates snapshots, or when snapshots contain generated IDs, timestamps, or random values that change on every run.

## Signal
- `jest --updateSnapshot` is aliased to `npm test` or run automatically in CI.
- Snapshot PRs show hundreds or thousands of changed lines across many files.
- Snapshots contain dynamic values: `id: "a3f7c..."`, `createdAt: "2024-01-15T..."`, `data-testid="random-123"`.
- Developers approve snapshot diffs without reading them.
- The same snapshot is updated 3+ times within a month.
- Full page-level snapshots that capture everything including navigation and footers for a single component test.

## Why
Snapshot tests are valuable when they capture the exact intentional output of a component or function and fail when that output changes unexpectedly. They become a maintenance burden — "change detectors" — when they are too large to review meaningfully, contain non-deterministic values, or are updated reflexively without considering whether the change is intentional. A snapshot that nobody reads and that is updated whenever a prop changes provides no more safety than not having it.

## Remediate

1. **Keep snapshots small and focused.** Snapshot a specific widget, not a full page. A snapshot test for a `<PriceTag price={9.99} currency="USD" />` component should capture only the rendered output of that component — not the parent layout with navigation and sidebar:
   ```tsx
   // Good — small, focused snapshot
   it('renders a price tag', () => {
     const { container } = render(<PriceTag price={9.99} currency="USD" />);
     expect(container).toMatchSnapshot();
   });
   // Result: a few lines of HTML, reviewable in seconds

   // Bad — full page snapshot
   it('renders the product page', () => {
     const { container } = render(<ProductPage id="123" />);
     expect(container).toMatchSnapshot();
     // Result: 400 lines of HTML capturing nav, sidebar, footer, etc.
   });
   ```

2. **Exclude dynamic values.** Any value that changes between runs (timestamps, UUIDs, random IDs, session tokens) must be stabilized before snapshotting:
   ```tsx
   // Option 1: mock the dynamic source
   jest.spyOn(crypto, 'randomUUID').mockReturnValue('static-id-for-tests');

   // Option 2: use Jest asymmetric matchers in serialized form
   expect(component).toMatchSnapshot({
     id: expect.any(String),
     createdAt: expect.any(String),
   });

   // Option 3: stable data-testid attributes (not autogenerated IDs)
   // Use data-testid="price-tag" not data-testid={`price-${uuid}`}
   ```

3. **Use inline snapshots for small, readable outputs.** `toMatchInlineSnapshot()` puts the snapshot directly in the test file, making it visible and reviewable in PR diffs without needing to open a separate `.snap` file:
   ```tsx
   it('renders a badge', () => {
     const { container } = render(<Badge count={5} />);
     expect(container).toMatchInlineSnapshot(`
       <div>
         <span class="badge">5</span>
       </div>
     `);
   });
   ```
   Inline snapshots are self-documenting and make reviewing intentional output changes obvious in the PR.

4. **Treat snapshot diffs as meaningful assertions in PR review.** In your PR review checklist: "snapshot changes must be reviewed and intentionally approved." Changes to snapshots should be accompanied by a description of *why* the output changed — not just "updated snapshots." A PR description that says "updated snapshots to match new design" is acceptable. One that says "ran --updateSnapshot" is not.

5. **Quarantine rapidly-changing snapshots.** If a snapshot is updated more than 3 times in a month, it is not stable enough to be a snapshot test. Replace it with explicit assertions about the specific properties that matter:
   ```tsx
   // Replace unstable full-output snapshot with explicit assertions
   const { getByText, getByRole } = render(<PriceTag price={9.99} currency="USD" />);
   expect(getByText('$9.99')).toBeInTheDocument();
   expect(getByRole('img', { name: /USD currency/i })).toBeInTheDocument();
   ```

6. **Never update snapshots in CI automatically.** CI should only run `jest --ci` (which fails on unexpected snapshot changes, never updates). Snapshot updates must be made manually by a developer who has understood the change.

7. **Review snapshot coverage.** Audit your `.snap` files: are they large or small? Do they test visual structure or business output? Large HTML snapshots rarely catch meaningful regressions because the signal is lost in the noise. Consider deleting large snapshots and replacing them with targeted assertions.
